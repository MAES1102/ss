How the Protection Works

The Problem: Fake Data

Without validation, any client can send arbitrary JSON. An attacker can change amount: 100 to amount: 99999 and the server has no way to detect it.

Digital Signature (RSA)

The sender hashes the data and encrypts the hash with a private key. The server verifies using the public key. Any modification — even one bit — produces a different hash → signature fails.

Encryption (AES-256-GCM)

Data is encrypted so it is unreadable in transit. GCM mode also produces an auth tag — if ciphertext is tampered, decryption throws an error before the data reaches application logic.

JWT Authentication

A JSON Web Token is a signed credential. The server issues it on login; every protected request must carry it. Expired, forged, or modified tokens are rejected at the middleware layer.

TLS Certificate (HTTPS)

This server runs over HTTPS using a self-signed X.509 certificate (RSA-2048). TLS encrypts the transport channel, preventing passive eavesdropping of JWTs and session data. All application-layer cryptography depends on this foundation.

Replay Protection (Nonce)

Each secure payload includes a UUID nonce embedded inside the signed and encrypted body. The server records accepted nonces and rejects duplicates, preventing retransmission of a previously valid captured request.
