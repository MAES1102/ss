## Why Each Improvement Matters

### Phase 2 — TLS / HTTPS

**Before:** All traffic travels over plain HTTP. A passive eavesdropper on the same network can read the JWT, the encrypted payload, and the signature with a single `tcpdump` or Wireshark capture.

**After:** TLS encrypts the transport channel. An eavesdropper sees only random ciphertext. The server's identity is attested by the certificate — a browser will warn the user if the certificate is invalid or substituted (MITM).

**Key concept:** The project topic says "conventional cryptographic tools." TLS *is* the most conventional and widely deployed cryptographic tool in existence. Without it, all application-layer cryptography (AES, RSA, JWT) is undermined because the AES key and JWT travel in the open.

**Why self-signed is still valid here:** A self-signed certificate provides the same TLS encryption as a CA-signed one. The difference is that a CA-signed cert proves identity to strangers. For a `localhost` demo where both client and server are on the same machine, self-signed is correct and sufficient.

### Phase 3 — Environment Variable Secrets

**Before:** `const JWT_SECRET = 'demo_super_secret_key_change_in_prod'` — visible to anyone who reads the source code, is committed to version control, or has read access to the file system.

**After:** Secret loaded from `.env` via `dotenv`. `.env` is in `.gitignore` — it never enters version control. The startup guard (`process.exit(1)`) ensures the server won't silently run with no secret.

**Why this matters:** Hardcoded secrets are one of the OWASP Top 10 vulnerabilities (A02:2021 — Cryptographic Failures). Even a 256-bit AES key is useless if it's in a public GitHub repository.

---

### Phase 4 — bcrypt Password Hashing

**Before:** `password: 'password123'` stored as plaintext. Comparison: `password !== DEMO_USER.password` — instant, constant-time, but completely insecure.

**After:** `passwordHash: bcrypt.hashSync('password123', 10)` — a salted, slow hash. Verification: `await bcrypt.compare(candidate, hash)`.

**Three properties of bcrypt that matter:**
1. **Salted:** Each hash includes a random 128-bit salt. Two users with the same password have different hashes → rainbow table attacks fail.
2. **Slow by design:** Cost factor 10 means ~100ms per comparison. Brute-forcing 1 billion passwords takes ~3 years on one CPU core.
3. **One-way:** The plaintext cannot be recovered from the hash even with the salt.

---

### Phase 5 — Nonce-Based Replay Protection

**Before:** A valid `/send-secure` request could be captured (e.g., from server logs, a packet capture, or a compromised network) and re-sent indefinitely. The server had no memory of previous requests. JWT + AES + RSA would all pass — the replay would be accepted.

**After:** Every secure payload includes a `nonce` (UUID v4 generated by `crypto.randomUUID()`). The nonce is embedded *inside* the signed and encrypted payload:
```json
{ "amount": 100, "from": "Alice", "to": "Bob", "nonce": "a4f3c2d1-..." }
```
The server stores accepted nonces in a `Set`. On the second request with the same nonce: **rejected**.

**Why the nonce is inside the signature:** If the nonce were outside the encrypted body, an attacker could strip it and substitute a fresh one. Because it is part of the signed payload, modifying it breaks the RSA signature.

**Why UUID is sufficient:** The probability of a UUID v4 collision is 1 in 2¹²² (~5×10³⁶). With 1 million requests per second, expected collision time is 2.7 × 10¹² years.

**Demo — Scenario 5:** Run Scenario 3, then immediately run Scenario 5. The server returns:
```json
{ "status": "rejected", "error": "Replay attack detected! This nonce has already been used." }
```

### Phase 6 — Restricted CORS

**Before:** `app.use(cors())` — accepts requests from ANY origin. A malicious website (`evil.com`) could make authenticated requests to the server using the victim's JWT stored in the browser.

**After:**
```javascript
app.use(cors({
  origin:         'https://localhost:3000',
  methods:        ['GET', 'POST'],
  allowedHeaders: ['Content-Type', 'Authorization'],
}));
```
Only requests originating from our own origin are accepted. This prevents cross-site request forgery (CSRF) at the CORS layer.

## Likely Professor Questions and Answers

### On TLS / Certificates

**Q: What is the difference between your self-signed certificate and a CA-signed certificate?**
> A: Both provide identical TLS encryption strength. The difference is trust establishment. A CA-signed certificate is issued by a Certificate Authority whose public key is pre-installed in all browsers (a "root of trust"). Browsers automatically trust it. A self-signed certificate is signed by its own private key — the browser has no prior knowledge of it and shows a warning. For a `localhost` demo where we control both endpoints, self-signed is correct. In production, we would use Let's Encrypt (free CA) to issue a trusted certificate.

**Q: Why is HTTPS necessary if you already have AES encryption in the payload?**
> A: AES encrypts the *payload*, but not the HTTP headers — including the `Authorization: Bearer <JWT>` header. Without TLS, a passive attacker captures the JWT from the header in plaintext. With that JWT they can call any protected endpoint as the authenticated user. TLS encrypts everything: headers, body, method, URL. The application-layer crypto and TLS are complementary, not redundant.

**Q: What does the `openssl req -x509` command actually do?**
> A: It generates a new RSA-2048 private key and a self-signed X.509 certificate in a single step. `-x509` means "output a self-signed certificate instead of a CSR." `-nodes` means "no DES" — don't encrypt the private key file. `-days 365` sets the validity period. `-subj "/CN=localhost"` sets the Common Name, which the browser checks against the hostname.

### On bcrypt

**Q: Why bcrypt and not SHA-256 for password hashing?**
> A: SHA-256 is a general-purpose hash — it can compute 10 billion hashes per second on a modern GPU. An attacker with a leaked hash database can brute-force common passwords in seconds. bcrypt's cost factor makes each computation take ~100ms on a CPU — the same attack now takes thousands of years. SHA-256 is the right tool for signing data; bcrypt is the right tool for passwords.

**Q: What is the salt in bcrypt?**
> A: A random 128-bit value generated fresh for each password. It is prepended to the hash and stored with it (the bcrypt output includes the algorithm identifier, cost factor, salt, and hash all in one 60-character string). The salt prevents two users with the same password from having the same hash, and makes pre-computed rainbow table attacks impossible.

---

### On Nonces and Replay Attacks

**Q: What is a replay attack?**
> A: An attacker captures a valid, authenticated HTTP request (e.g., via a network packet capture) and re-sends it to the server later. Even though the attacker cannot read or modify the contents (AES + RSA protect that), the server cannot distinguish a replayed request from a fresh one — it has no memory. Without nonce protection, replaying `{ amount: 100, from: "Alice", to: "Bob" }` 100 times would process 100 transactions.

**Q: Why is the nonce inside the encrypted payload rather than in an HTTP header?**
> A: If the nonce were a plain HTTP header, an attacker could strip the old nonce and replace it with a fresh one — the server would accept it as a new request, and the replay would succeed. By embedding the nonce inside the RSA-signed payload, any modification — including nonce substitution — breaks the signature. The nonce's freshness is cryptographically bound to the data's integrity.

**Q: What happens when the server restarts — are all nonces forgotten?**
> A: Yes. The `usedNonces` `Set` is in-memory. After a restart, a previously accepted nonce could be replayed within the original JWT's 1-hour window. In production this is solved with Redis (persistent, TTL-aware key-value store) where each nonce entry expires after the JWT expiry time.

---

### On CORS

**Q: What is a CSRF attack and how does CORS restriction prevent it?**
> A: Cross-Site Request Forgery (CSRF) is when a malicious page at `evil.com` makes a request to `localhost:3000` using the victim's active session cookies or tokens. With `cors()` (permissive), the browser would include the credentials and the server would accept. With `cors({ origin: 'https://localhost:3000' })`, the browser enforces the Same-Origin Policy and the preflight `OPTIONS` request is rejected — the actual request never reaches the server.
